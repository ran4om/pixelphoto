## Screen Navigation Pattern

```typescript
type Screen = 'home' | 'config' | 'run' | 'presets' | 'onboarding'

class App {
  currentScreen: Screen = 'home'
  
  navigate(screen: Screen) {
    this.currentScreen = screen
    this.render()
  }
  
  render() {
    renderer.root.clear()
    switch (this.currentScreen) {
      case 'home': return renderHome(this)
      case 'config': return renderConfig(this)
      case 'run': return renderRun(this)
      case 'presets': return renderPresets(this)
      case 'onboarding': return renderOnboarding(this)
    }
  }
}
```

# OpenTUI + Ghostty stdin Parsing Fix Plan

## Problem Summary

Ghostty batches multiple escape sequences into a single stdin chunk. OpenTUI incorrectly assumes one chunk = one type of input, causing:
- Raw escape sequences printed as garbage text
- Mouse input broken or unreliable  
- Keyboard input misrouted

## Root Cause (CRITICAL)

OpenTUI's stdin handling in renderer.ts has a broken dispatch:

```
stdin chunk arrives: "\x1b[<0;123;456M\x1b[?2026h\x1b[<0;124;457M"
  │
  ├─► Does chunk START with mouse data? ──► handleMouseData() ──► Returns, DROPS everything else!
  │
  └─► Otherwise ──► _stdinBuffer.process() ──► Mouse sequences inside NEVER reach MouseParser
```

This is documented in OpenTUI issue #758. The problem is at the CHUNK level, not the sequence handler level.

**Even with prependInputHandlers**, if the chunk starts with mouse data, the subsequent sequences are dropped entirely and never reach any handler.

### Additional Issues

1. Multiple independent parsers (MouseParser, StdinBuffer, inputHandlers, TerminalPalette) don't coordinate
2. `autoFocus: true` triggers re-render mid-sequence, desyncs hit grid
3. Mouse mode escape sequences not properly cleaned up on exit

## Solution Strategy

### Challenge

We can't easily replace OpenTUI's stdin handling because:
1. The stdin listener is internal to CliRenderer
2. The StdinParser is private 
3. We can't intercept raw chunks before they reach the broken dispatcher

### Available Options

**Option A: prependInputHandlers (partial fix)**
- Pros: Built-in OpenTUI feature, easy to use
- Cons: May not help if sequences are dropped BEFORE handlers are called

**Option B: Replace stdin stream entirely (complex)**
- Pros: Full control over parsing
- Cons: Requires deep OpenTUI internals knowledge, may break on updates

**Option C: Disable problematic features + improve parser (recommended)**
- Reduce stdin complexity by disabling features that cause batching
- Use prependInputHandlers for debug/logging
- Fix autoFocus and signal handling

      createInputHandler(renderer),
    ],
  });

  // Explicit SIGINT/SIGTERM handling - prevents mouse mode leak
  const cleanupAndExit = () => {
    // Disable mouse modes before exit
    if (renderer.stdout && renderer.stdout.write) {
      renderer.stdout.write('\x1b[?1003l'); // Disable movement tracking
      renderer.stdout.write('\x1b[?1002l'); // Disable button tracking
      renderer.stdout.write('\x1b[?1000l'); // Disable mouse tracking
      renderer.stdout.write('\x1b[?1006l'); // Disable SGR extended
    }
    renderer.destroy();
    process.exit(0);
  };
  
  process.on('SIGINT', cleanupAndExit);
  process.on('SIGTERM', cleanupAndExit);

  // ... rest of setup ...
}

#### 3. stdinParser.ts - Debug and monitoring

```typescript
// src/tui/stdinParser.ts

import { MouseEventType, RawMouseEvent, ScrollInfo } from '@opentui/core';

const DEBUG = process.env.DEBUG_STDIN === '1';

function debug(...args: any[]) {
  if (DEBUG) console.error('[stdin-parser]', ...args);
}

export interface StdinToken {
  type: string;
  raw: string;
  data?: any;
}

/**
 * Streaming stdin parser for debugging and monitoring.
 * Extracts escape sequences from potentially batched stdin chunks.
 */
export class StdinParser {
  private buffer: string = '';
  
  processChunk(chunk: string): StdinToken[] {
    debug('Chunk:', JSON.stringify(chunk));
    this.buffer += chunk;
    const tokens: StdinToken[] = [];
    
    while (this.buffer.length > 0) {
      const char = this.buffer[0];
      
      if (char === '\x1b') {
        const seq = this.extractSequence(this.buffer);
        if (seq) {
          tokens.push(this.classify(seq));
          debug('Extracted:', seq);
          this.buffer = this.buffer.slice(seq.length);
        } else {
          break;
        }
      } else {
        tokens.push({ type: 'text', raw: char });
        this.buffer = this.buffer.slice(1);
      }
    }
    
    return tokens;
  }
  
  private extractSequence(buf: string): string | null {
    // SGR Mouse: \x1b[<button;x;yM or \x1b[<button;x;ym
    if (buf.match(/^\x1b\[<\d+;\d+;\d+[Mm]/)) {
      const match = buf.match(/^\x1b\[<\d+;\d+;\d+[Mm]/);
      return match![0];
    }
    
    // CSI sequences
    if (buf.startsWith('\x1b[')) {
      const match = buf.match(/^\x1b\[[\x30-\x3F]*[\x20-\x2F]*[\x40-\x7E]/);
      return match ? match[0] : null;
    }
    
    // OSC sequences  
    if (buf.startsWith('\x1b]')) {
      const match = buf.match(/^\x1b\][^\x07\x1b]*(?:\x07|\x1b\\)/);
      return match ? match[0] : null;
    }
    
    return null;
  }
  
  private classify(seq: string): StdinToken {
    if (seq.match(/^\x1b\[<\d+;\d+;\d+M$/)) {
      return { type: 'mouse-press', raw: seq };
    }
    if (seq.match(/^\x1b\[<\d+;\d+;\d+m$/)) {
      return { type: 'mouse-release', raw: seq };
    }
    if (seq.match(/^\x1b\[<6[45];\d+;\d+M$/)) {
      return { type: 'mouse-scroll', raw: seq };
    }
    return { type: 'keyboard', raw: seq };
  }
  
  reset() {
    this.buffer = '';
  }
}

export function createInputHandler(renderer: any) {
  const parser = new StdinParser();
  
  return (sequence: string): boolean => {
    const tokens = parser.processChunk(sequence);
    
    // Log for debugging - helps identify what's arriving
    debug('Tokens:', tokens.map(t => t.type).join(', '));
    
    // Return false to let OpenTUI's default handlers process
    // Returning true would consume the sequence entirely
    return false;
  };
}
```

## Why This Might Not Fully Fix It

The core issue is in OpenTUI's internal stdinListener which we can't easily intercept. The `prependInputHandlers` receive sequences AFTER they've been extracted by the (potentially broken) internal parser.

If sequences are being dropped at the chunk level (before sequence extraction), our handler will never see them.

## Fallback: More Aggressive Disabling

If the above doesn't work, we may need to:

1. Disable mouse entirely (`useMouse: false`) - sacrifices mouse but keyboard works
2. Use `useMouse: 'cell'` if OpenTUI supports it - alternative mouse protocol
3. Submit bug report to OpenTUI with detailed reproduction steps
